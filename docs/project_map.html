2. Image Generation lifecycle

01

Authentication & Identity

The app verifies the user via Firebase (Google) or Email/Password and issues a Bearer JWT.

POST /auth/google
GET /auth/me
02

Credit Check & Refresh

Before generation, the system ensures daily_credits are refreshed and total balance ≥ 1.

Priority: Daily > Purchased
Reset: 00:00 UTC
03

Style Selection & Upload

User picks a Style (e.g. Ghibli). The original photo is streamed to an S3 "Originals" bucket.

Multipart/Form-Data
AWS S3 Signed URLs
04

AI Transformation (Gemini)

The backend merges the style_template with user mood and sends bits to Gemini Pro Vision.

Google Generative AI SDK
Prompt Engineering
05

Persistence & Delivery

The result is saved to S3. Credits are deducted, and the Creation ID is returned.

DB: creation_id
Feed: is_public check
06

Secondary Economy: AdMob Rewards

Users can watch videos to earn free credits. The backend enforces strict daily limits (e.g., 5 ads/day).

POST /rewards/admob
Audit: credit_transactions

🔐 Firebase Google Logic

We don't trust client-side claims. The app sends a tokens.google.id_token. Our backend verifies the Audience (aud) against our Android Client ID to prevent token spoofing from web apps or other projects.

💳 Credit Spending Rule

Economy logic: Daily free credits are ephemeral. We use min(daily_credits, cost) first. This incentivizes users to use the app daily while preserving their paid balance.

🚀 Performance

FastAPI handles async S3 uploads while the Gemini AI processes in a separate thread, ensuring the API doesn't hang under high concurrent load.

📺 AdMob Architecture

The system uses a Rewarded Ad flow:

  • Mobile: Displays ad via AdMob SDK.
  • Callback: Upon completion, onUserEarnedReward triggers a backend hit.
  • Backend: Reads DAILY_AD_WATCH_LIMIT from DB to prevent botting/farming.
  • Ledger: Atomically updates user balance and writes to ad_watches for audit.
